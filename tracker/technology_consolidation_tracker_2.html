Why not AWS CodeCommit?

A fair comparison — CodeCommit is a real option again, and here’s why GitHub is still the stronger foundation.

Correcting the record: AWS de-emphasized CodeCommit in July 2024, closing it to new sign-ups. AWS then reversed course and returned it to full General Availability on November 24, 2025 — new customers can sign up again, and AWS says it’s investing in the roadmap (Git LFS in Q1 2026, new regions in Q3 2026). It is not a dead product. The case below stands on its own merits regardless.

Back in the running

AWS CodeCommit

A credible, governed option with real strengths — deep IAM integration, VPC endpoints, and CloudTrail logging that appeal directly to a regulated bank. But it replaces exactly one AWS DevOps line item: repo hosting. CodeBuild, CodePipeline, CodeArtifact, code review tooling, and a dev environment would still be separate services we configure and wire together ourselves. And AWS's own 16-month wobble on the product — de-emphasized in July 2024, reinstated in November 2025 — is a fair data point when picking the platform a growing department builds on for years.

The industry standard

GitHub Enterprise

Repos, Actions (CI/CD), Packages, Codespaces cloud dev environments, and Advanced Security in one governed platform — one Entra sign-in, one audit log, one bill. Per GitHub, 100M+ developers and roughly 90% of the Fortune 100 build here. Almost every developer we hire already knows it on day one.

This isn’t anti-AWS — our workloads stay on AWS, and CodeCommit is a legitimate product again. It’s about not assembling nine separate DevOps services by hand when one platform ships them governed, out of the box.

What is Advanced Security, really?

The add-on behind the big third bar — what it does, what it costs, and the honest alternatives if we skip it. Pick a card, or compare all four.

Gap 2 · Agentic AI development

Copilot assists. It doesn’t do the work. That’s the gap.

Modern development teams hand whole tasks to a coding agent — it navigates the repo, writes and tests the change, and drafts the documentation, while the developer moves to the next problem. Microsoft 365 Copilot is not that tool and was never meant to be. Claude Enterprise, deployed through Entra ID with full audit logging, is — and it’s complementary to Copilot, not a replacement.

Recommendation: Adopt Claude Enterprise for the technology organization's power users, with Claude Code (and its VS Code extension) as the primary use case. Why not just Microsoft Copilot?

This is a right-tool-for-the-job decision, not a verdict on Microsoft. The two are complementary.

Firm-wide default

Microsoft 365 Copilot

The right regulated assistant for everyone: drafting, summarizing, analyzing documents, and working inside Office and Teams. This decision does not replace it — Copilot should be the assistant in front of all end users.

Coding specialist

Claude Code

An agent that takes a whole task: navigate a repository, read and edit across many files, run commands and tests under explicit permission — from the terminal, VS Code, or desktop. That whole-task capability is what a general chat assistant or autocomplete tool does not provide.

Copilot for everyone; Claude for the people who build and secure our software.

What an agent changes day to day

Concrete workflow wins — hours back per developer per week, not abstract AI promise.

Delegate

Hand off whole tasks, not lines

Claude Code takes a task — “upgrade this dependency and fix what breaks,” “add tests around this module” — works across many files, runs the tests under explicit permission, and reports back. The developer reviews a finished change instead of typing one, and works on something else in the meantime.

Document

Documentation stops being procrastinated

The agent reads the diff and the repo and drafts elegant docs that fit our structure and hit the important details — the developer edits instead of authoring from a blank page. This is the engine behind the Gap 4 documentation proposal.

Recall

Total recall of the codebase

An agent references thousands of files, past patterns, and team conventions in seconds — the “where is this handled?” and “what breaks if I change this?” questions that eat senior developers’ time stop being interruptions.

Resolve

Issues get solved faster

Point the agent at a failing test, a stack trace, or a ticket and it traces the cause across the codebase before proposing the fix — shrinking the debug loop that dominates real development time.

A second AI stack? — Jason’s note

Raised by Jason on the team: don’t build a one-vendor AI strategy. Worth discussing, not deciding, here.

Jason’s note · team input

Run 2+ AI stacks — and evaluate xAI as the second

The suggestion: keep at least two independent AI stacks so we're never single-vendor on the most strategically important tooling we're adopting. Jason's candidate for the second stack is xAI — recent articles he's been reading paint it as substantially cheaper with strong performance on most coding workloads. xAI now ships Grok Build (x.ai/cli), a terminal-native agentic coding CLI in the same category as Claude Code: plan-first execution, parallel sub-agents in git worktrees, and native MCP support.

The bar it would have to clear: exactly the one this tab holds Claude to — Entra SSO/SCIM, exportable audit logs, retention we control, and whether it can tie into Varonis the way Claude Enterprise and Copilot already do. That Varonis question is open: no equivalent of Anthropic's Compliance API integration has been confirmed for xAI, so it's the first thing to diligence. Reported pricing (unverified, changes fast): Grok Build rides SuperGrok/X Premium+ subscriptions (~$30–$300/mo tiers), with API token rates well below Anthropic's list — treat those claims as Jason's reading, not vetted figures, until we run the same due-diligence pass we ran on Claude.

Why it's practical now

MCP makes a two-stack strategy cheap to run

The plumbing is already portable. MCP standardizes the tool layer — one MCP server (GitHub, Jira, our internal APIs) plugs into any MCP client, and Grok Build is designed to read the same .mcp.json config Claude Code uses, with zero reconfiguration. Shared instructions live as plain files in the repo (skills / AGENTS.md), so both agents follow the same playbook. What can't be standardized is the models' judgment — consistency comes from objective ground truth: the test suite, linters, and CI checks both agents are graded against, plus human review at the merge gate.

The honest caveats

What we'd verify before adopting

(1) Governance parity — SSO, SCIM, audit export, retention, and a Varonis path; without those it fails the same test every non-Enterprise Claude tier fails. (2) MCP transport — Grok Build currently accepts only Streamable HTTP/SSE servers, not stdio; our local-style MCP servers would need a tunnel or hosted variant to reach it, even though Claude Code talks to them directly. (3) Pricing volatility — the cost claims are article-sourced; get list pricing and an enterprise quote in writing. None of these are blockers — they're the checklist.

One item to flag before rollout

The single caveat in an otherwise clean governance story — parked here so it doesn’t distract from the case above.

Disable Claude Cowork at launch (or scope it to a small group). Cowork session activity is currently excluded from Anthropic's audit logs, Compliance API, and data exports on every tier, and its history is stored on the local device — which conflicts with our audit and retention requirements. Our primary use case, Claude Code, is fully covered by audit logging, so this costs us nothing today.

Why this is a productivity multiplier

This is not simply a more modern ticketing system; it removes the manual work that slows the team down today.

Auto-created

Tickets are created automatically

Alerts from Defender, Rapid7, and Proofpoint open tickets automatically; a GitHub pull request can open and update its own ticket; portal requests are routed automatically by category, priority, and location. The queue populates on its own, freeing technicians to triage and resolve rather than manually log tickets.

Always documented

Every change is documented as it happens

The moment a change is raised, JSM captures who, what, when, the approvals, and the risk level in one ITIL-aligned record. Nothing ships undocumented, and the audit trail is built continuously as work happens rather than reconstructed ahead of an examination.

Assembled for you

Change Management packets compile automatically

The weekly CAB packet — pending changes, approvals, risk ratings, and post-implementation reviews — is generated from the system of record rather than compiled manually in a separate document. The meeting begins from live data.

Deflected

Routine requests resolve through self-service

A modern self-service portal backed by a Confluence knowledge base resolves the repeat requests — password resets, access provisioning, how-tos — before they reach a technician. First-call resolution improves while the manual queue shrinks.

Change Management, automated end to end

Where the manual overhead is heaviest today — and where JSM removes the most work.

The weekly Change Management meeting stops being a manual assembly job. Every change raised in JSM carries its approvals, risk assessment, and post-implementation review with it — so the CAB review and the monthly ISC report are generated from live records rather than rebuilt by hand each week.

What Docusaurus gives us out of the box

Not a side project — it’s the docs framework behind React, Jest, and hundreds of major open-source projects.

  • Navigation builds itself — sidebar and nav auto-generate from the folder structure; a new .md file just shows up, no manual config.
  • Table of contents for free on every page, auto-built from headings and synced to scroll position.
  • Code blocks done right — syntax highlighting, line numbers, titles, one-click copy, and line-range highlighting for walkthroughs.
  • Mermaid diagrams in Markdown — architecture and flow charts live in the doc, versioned with it.
  • Interactive when needed — MDX lets us drop React components into a page when plain Markdown isn’t expressive enough.
  • Tabs & admonitions built in — per-OS setup instructions, and callouts like tip / caution / danger.
  • Dark mode out of the box, themable to bank colors with plain CSS.
  • Doc versioning — keep v1 vs. v2 snapshots of an internal API when history matters.
  • Search across everything — a full search experience over all our docs, out of the box.
  • Nothing to maintain — a static build with no server to patch; SEO/metadata handled via front matter if anything ever goes external.

First, the AWS-native reality in 2026

The obvious “just use AWS” answer is narrower than it was — worth stating plainly before we compare.

AWS has effectively exited the managed cloud-IDE business for new customers. Cloud9 was closed to new sign-ups on July 25, 2024. Its intended successor, Amazon CodeCatalyst, was itself closed to new customers on November 7, 2025 and moved to maintenance mode — no new features, and existing customers can’t even create new spaces. So the AWS-native path today is not a managed product we can adopt; it’s either self-managed dev containers on raw EC2, or — more practically — running Coder or Ona on our AWS. That reframes AWS’s role here as the infrastructure underneath, not the IDE itself.
